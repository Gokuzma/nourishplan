## Phase Details

### Phase 16: Budget Engine & Query Foundation
**Goal**: Households can track ingredient costs, see cost per recipe serving, and monitor weekly spend against a household budget — and all v2.0 queries share a centralised key hierarchy preventing cache incoherence.
**Requirements**: BUDG-01, BUDG-02, BUDG-03, BUDG-04
**Plans**: 4 (4/4 complete)

### Phase 17: Inventory Engine
**Goal**: Household members can maintain a pantry/fridge/freezer inventory with quantities, units, expiry dates, and storage locations — and the inventory updates automatically when a meal plan is finalised or leftovers are logged.
**Requirements**: INVT-01, INVT-02, INVT-03, INVT-04, INVT-05, INVT-06
**Plans**: 5 (4/5 complete — Plan 05 retrospective verification)

### Phase 18: Grocery List Generation
**Goal**: The app auto-generates a categorised grocery list from the active meal plan, subtracts what the household already has in inventory, and lets household members check off items while shopping.
**Requirements**: GROC-01, GROC-02, GROC-03, GROC-04, GROC-05
**Plans**: 3 (3/3 complete)

### Phase 19: Drag-and-Drop Planner
**Goal**: Users can rearrange meals on the weekly plan grid by dragging and dropping, and manually placed meals are locked so auto-generation cannot overwrite them.
**Requirements**: PLAN-01, PLAN-03
**Plans**: 3 (3/3 complete)

### Phase 20: Feedback Engine & Dietary Restrictions
**Goal**: Household members can rate recipes, flag satiety, set dietary restrictions, list foods they won't eat — and the system warns when the plan becomes monotonous.
**Requirements**: FEED-01, FEED-02, FEED-03, FEED-04
**Plans**: 4 (4/4 complete)

### Phase 21: Schedule Model
**Goal**: Each household member can set their daily availability for meal prep and eating, stored as structured constraints ready to feed the Planning Engine.
**Requirements**: SCHED-01, SCHED-02
**Plans**: 3 (3/3 complete)

### Phase 22: Constraint-Based Planning Engine
**Goal**: The app can generate a complete weekly meal plan optimised across nutrition targets, household budget, member schedules, dietary restrictions, and recipe preference signals — without blocking the UI.
**Requirements**: PLAN-02, PLAN-04, PLAN-05
**Plans**: 9 (9/9 complete, including 4 gap-closure plans 22-06..09)

### Phase 23: Prep Optimisation
**Goal**: Users can see a batch prep schedule for the week and a day-of task sequence for any meal, so cooking time is used efficiently.
**Requirements**: PREP-01, PREP-02, PREP-03
**Plans**: 10 (10/10 complete)

### Phase 24: Dynamic Portioning → Tier-Aware Recipe Selection (D-02 pivot)
**Goal (shipped scope)**: Tier-aware recipe selection via `generate-plan` edge function enrichment + RecipeMixPanel three-slider component. Portions remain calorie-target-driven (PORT-01).
**Requirements**: PORT-01, PORT-02
**Plans**: 2 (2/2 complete)
**Note**: Original D-02 scope (satiety-adaptive portion-size adaptation) pivoted mid-phase — rationale in `.planning/phases/24-dynamic-portioning/24-CONTEXT.md`.

### Phase 25: Universal Recipe Import
**Goal**: Users can paste a URL (blog, YouTube) or raw recipe text and the AI extracts a complete recipe with ingredients, macros, and instructions — saved directly to the recipe library.
**Requirements**: IMPORT-01, IMPORT-02, IMPORT-03, IMPORT-04, IMPORT-05
**Plans**: 3 (3/3 complete)

## Milestone Summary

### Key Accomplishments

1. **Constraint-solving Planning Engine** — async `generate-plan` edge function optimises across nutrition, budget, schedule, dietary restrictions, tier weighting, and inventory priority
2. **End-to-end inventory lifecycle** — pantry/fridge/freezer with ledger-based quantities, FIFO deduct on cook, leftover save flow, barcode scanning
3. **Household-shared grocery list** with realtime sync, pantry subtraction, store-aisle categorisation, and in-store check-off
4. **Drag-and-drop weekly planner** with locked-slot preservation (dnd-kit), schedule-badge overlays
5. **Universal recipe import** — URL + paste-text (+ YouTube transcript extraction) via `import-recipe` edge function
6. **Granular household roles** — admin/member RPCs, DB-enforced last-admin protection, invite-time role selection, first Playwright E2E regression test in the repo

### Technical Decisions

See PROJECT.md "Key Decisions" table. Highlights:
- D-02 pivot (Phase 24): tier-aware recipe selection instead of dynamic portioning
- Binary admin/member (Phase 30): reuses 17 existing RLS policies without rewrites
- Last-admin protection at DB trigger level (Phase 30)
- Retrospective VERIFICATION.md pattern (Phase 29) for already-shipped phases
