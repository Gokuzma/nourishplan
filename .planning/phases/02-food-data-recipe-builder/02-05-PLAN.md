**Nested recipe support in RecipeBuilder.tsx:**
- When FoodSearch is open for adding ingredients, add a fourth option: "Add Recipe as Ingredient". This could be a button below the FoodSearch or a tab within FoodSearch. Simpler approach: add a "Recipes" section below the FoodSearch overlay that lists household recipes (using `useRecipes()`). Each recipe row is selectable.
- When a recipe is selected as ingredient: BEFORE adding, run `wouldCreateCycle(currentRecipeId, selectedRecipeId, getIngredientsForCycle)` where `getIngredientsForCycle` fetches recipe_ingredients from Supabase for a given recipe ID.
- If cycle detected: show inline error message "Cannot add this recipe — it would create a circular reference" and do NOT add the ingredient.
- If no cycle: prompt for quantity (grams), then call `useAddIngredient({ recipe_id, ingredient_type: 'recipe', ingredient_id: selectedRecipeId, quantity_grams })`.
- For nested recipe ingredients, resolve nutrition by recursively fetching the sub-recipe's ingredients and calculating. Use memoization to avoid refetching. Store resolved nutrition in the local `ingredientFoodData` map.

**IngredientRow.tsx — recipe badge and nested view:**
- Per user decision: nested recipes show inline with a recipe badge icon (a small icon like a book or utensil before the name)
- Per user decision: tappable to view sub-recipe — clicking the name navigates to `/recipes/${ingredient_id}` (or opens in a sheet/modal — simpler to navigate for v1)

**Raw/cooked toggle in IngredientRow.tsx:**
- Per user decision: raw/cooked toggle per ingredient row, default is raw
- Add a small toggle button or segmented control ("Raw" | "Cooked") on each IngredientRow
- When toggled: call `useUpdateIngredient()` to update `weight_state`
- Nutrition recalculation: in RecipeBuilder, when calculating nutrition for an ingredient with weight_state === 'cooked', use `applyYieldFactor(quantity_grams, 'cooked', YIELD_FACTORS[foodCategory] || 1.0)` to get the equivalent raw grams, then calculate nutrition from that.
- Food category for yield factor: for v1, use a simple heuristic — store no category, use a default factor of 0.85 (general cooking loss). The YIELD_FACTORS map is available for future per-category assignment. This is Claude's discretion.
- Per user decision: "toggling recalculates nutrition using weight conversion factors"

**AI verification integration in FoodSearch.tsx (light touch):**
- When USDA or OFF results are displayed, if verify-nutrition Edge Function is available, call it in the background for displayed results (batch the top 5 results)
- Per user decision: "AI verification runs at search time — results are pre-verified before the user sees them"
- Show small info icon (i in circle) on results that were AI-verified. Tapping shows a tooltip/popover with the verification reason.
- Show warning badge on results with outlier warnings. Tapping shows warning text.
- If AI verification fails or is unavailable, results still show without badges — graceful degradation.
- Implementation: add an effect in FoodSearch that triggers after search results load, calls `supabase.functions.invoke('verify-nutrition', { body: { foodName, usdaValues/offValues } })` for each result, and stores verification state alongside results.
