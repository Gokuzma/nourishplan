## Design System

| Property | Value |
|----------|-------|

The 7 tracked micronutrients each use a distinct `ProgressRing` color from the existing pastel palette. These are display-only — never used as interactive affordances.

| Nutrient | Color | Hex |
|----------|-------|-----|
| fiber | sage (matches calories ring) | `#A8C5A0` |
| sodium | slate blue | `#A2C4D8` |
| calcium | warm lavender | `#C4B4D8` |
| iron | dusty rose | `#D8A2A2` |
| potassium | muted gold | `#D8C8A2` |
| vitamin_c | pale citrus | `#C8D8A2` |

These colors are passed as the `color` prop to the existing `ProgressRing` component. They must be hardcoded constants in the component that renders the micronutrient summary (not tokens).

## Component Contracts

### CALC-01: Live Nutrition in RecipeBuilder and FreeformLogModal

**RecipeBuilder nutrition bar** — no visual changes. The fix is internal state propagation only. The `NutritionBar` component and its layout are unchanged.

**FreeformLogModal nutrition preview** — no visual changes. The displayed preview already uses `scaledMacro()` inline. The fix changes how `servings_logged` and per-serving macros are computed at insert time.

**EditLogModal macro preview** — already shows live macro values scaled by `servings`. The fix ensures the underlying data model supports correct scaling. The four-cell macro grid (`calories / protein / carbs / fat`) is unchanged.

### CALC-02: Micronutrient Summary on HomePage

A new section is added to `HomePage` between the macro rings card and the action buttons. It is always visible (not collapsed) and shows all 7 tracked micronutrients regardless of whether the user has set targets.

**Layout:**
```
[Macro rings card]          ← existing, unchanged
[Micronutrient summary]     ← NEW: inserted here
[Action buttons row]        ← existing, unchanged
[Daily log list]            ← existing, unchanged
[Nutrient Details (collapsed)] ← existing, unchanged
```

**Micronutrient summary card structure:**
- Container: `bg-surface rounded-[--radius-card] border border-secondary p-4`
- Section label: `text-xs font-semibold text-text/50 uppercase tracking-wide mb-3` — copy: "Micronutrients"
- Grid: 7 mini rings displayed in a 4+3 or scrollable horizontal row, matching the macro rings visual rhythm
- Each mini ring: `ProgressRing` at `size={40}` (existing mini-ring size from SlotCard), `strokeWidth={3}`
- Below each ring: nutrient label in `text-xs text-text/50`, e.g. "fiber", "sodium", "calcium"
- Rings show value/target progress. If no target is set (`target === null` or key absent from `target.micronutrients`), ring renders with `target={0}` — displays empty ring (0% fill), which is the existing `ProgressRing` behavior for `target > 0` guard (`pct = 0` when `target === 0`)

**Incomplete data warning:**
- Shown inline below the 7 rings when `hasIncompleteMicroData === true` (any log has empty `micronutrients` object)
- Styling: `text-xs text-text/40 mt-2`
- Copy: `* Some foods have no micronutrient data — totals may be incomplete`
- No icon. No banner. Inline note only. Do not show when all logged foods have micronutrient data.

**Value display format:** `{value}{unit}` with no space before unit for g/mg, space for mcg. Use `MICRONUTRIENT_UNITS` constants. Example: `24g`, `450mg`, `80mcg`.

**Detail expansion:** Tapping the "Micronutrients" label does not expand or collapse anything in the summary section. The existing `NutrientBreakdown` component (labelled "Nutrient Details") below remains the detail view. No duplication of expand/collapse behavior.

### CALC-03: Serving Unit in LogEntryItem

**LogEntryItem** serving line changes from:
```
1 serving  /  2 servings
```
to:
```
1.5 cups  /  200 g  /  2 tbsp  /  1 serving (legacy fallback)
```

The serving line is the second line of the main content area (`text-xs text-text/50`). Format: `{servings_logged} {serving_unit ?? 'serving'}`. No pluralization logic for unit names (units from USDA/CNF are already singular descriptors like "cup", "tbsp", "g"). The word "serving" is the only legacy fallback and should not be pluralized either (i.e., "2 serving" is acceptable for legacy entries — avoid adding pluralization complexity for a legacy edge case).

**FreeformLogModal** — no visual changes. The `serving_unit` value is passed at insert time from the selected `PortionUnit.description`. The existing unit selector UI is unchanged.

## Interaction Contracts

### Live nutrition updates (CALC-01)

- RecipeBuilder: Nutrition bar updates after `updateIngredient.mutate` success triggers TanStack Query re-fetch. No loading spinner on the nutrition bar — the re-fetch is fast enough that the existing stale-while-revalidate behavior is sufficient.
- FreeformLogModal: Nutrition preview updates on every keystroke in the quantity input (existing `scaledMacro()` inline render pattern). No change to this interaction.
- EditLogModal: Macro preview cells (`calories / protein / carbs / fat`) update live as `servings` state changes (already implemented — `log.calories_per_serving * servings`). After the fix lands, this calculation will produce correct values because `calories_per_serving` will hold true per-unit macros.

### Micronutrient summary (CALC-02)

- Summary section is always rendered when `HomePage` renders — no loading state needed (data comes from the same `logs` and `target` already fetched).
- Rings animate in using `ProgressRing`'s existing SVG `strokeDashoffset` approach — no additional animation needed.
- Empty state (no logs yet): All 7 rings show 0% fill. No empty state copy — the rings speak for themselves.

### Serving unit display (CALC-03)

- `LogEntryItem` is a pure display component — no interaction changes.
- `FreeformLogModal` unit selector interaction is unchanged (existing `PortionStepper` and unit dropdown).

## Accessibility Contract

- New micronutrient rings: each `ProgressRing` SVG does not require an `aria-label` — the text label directly below serves as the accessible label. The rings are decorative when paired with their label.
- Incomplete data warning: plain text, no role needed.
- `LogEntryItem` `aria-label` prop includes item name only — serving unit text is in the visual label, not the aria-label. No change to the aria-label pattern.

---

## Dark Mode Contract

All new elements use only theme tokens (`text-text`, `bg-surface`, `border-secondary`, `text-primary`) or opacity modifiers (`text-text/50`, `bg-secondary/20`). No hardcoded light-mode hex values in new code. The micronutrient ring colors (listed above) are hardcoded hex values passed as the `color` prop to `ProgressRing` — these are intentional brand colors that do not need dark-mode variants (they are stroke colors on a transparent background, not backgrounds themselves). This matches the existing `ringColors` pattern in `HomePage.tsx`.
