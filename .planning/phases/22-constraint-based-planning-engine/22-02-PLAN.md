Create `supabase/functions/generate-plan/index.ts` following the exact same Deno edge function pattern as `classify-restrictions/index.ts`.

**Request shape:**
```typescript
interface GenerateRequest {
  householdId: string
  planId: string
  weekStart: string
  priorityOrder: string[]  // e.g. ["Nutrition","Preferences","Budget","Variety","Inventory"]
}
```

**Function flow:**

1. **CORS + parse request** — same OPTIONS handling as classify-restrictions.

2. **Auth validation** — Extract user from JWT via `adminClient.auth.getUser(req.headers.get("Authorization")?.replace("Bearer ", ""))`. Verify user is a member of the requested household by checking `household_members` table. Return error if not a member (per T-22-01 security threat).

3. **Rate limiting** — Query `plan_generations` for this household where `created_at > now() - interval '24 hours'`. If count >= 10, return `{ success: false, error: "Rate limit exceeded. Maximum 10 generations per 24 hours." }`.

4. **Create job row** — INSERT into `plan_generations` with `status: 'running'`, `household_id`, `plan_id`, `triggered_by: user.id`, `priority_order`. Return `{ success: true, jobId: row.id }` to the client **immediately after this insert** (do not wait for AI).

5. **Background processing** — Use `EdgeRuntime.waitUntil` or a fire-and-forget async pattern (wrap the AI loop in a try/catch that updates the job row on error). The response has already been sent in step 4.

   If `waitUntil` is not available in the Supabase Deno runtime, use the simpler pattern: do NOT return early. Instead, run the entire flow synchronously but within the time budget. The client will poll the job row.

   **Revised approach (simpler, proven):** Run the full generation synchronously. The client gets the jobId in the response. The function continues running. When the function completes (or times out), the job row is updated. The client polls `plan_generations` by jobId.

6. **Constraint snapshot assembly (parallel)** — Record `const startTime = Date.now()`. Fetch ALL constraint data in parallel using `Promise.all`:
   - `recipes` with `recipe_ingredients` (household_id match, deleted_at IS NULL)
   - `dietary_restrictions` (household_id match)
   - `wont_eat_entries` (household_id match)
   - `member_schedule_slots` (household_id match)
   - `inventory_items` (household_id match, removed_at IS NULL)
   - `recipe_ratings` (household_id match)
   - `nutrition_targets` (household_id match)
   - `household_members` + `member_profiles` (household_id match)
   - `meal_plan_slots` (plan_id match) — to identify locked slots

   Serialize the snapshot to a `constraintSnapshot` object. Store a summary (not full data) in `plan_generations.constraint_snapshot`.

7. **Recipe count check** — If household has fewer than 7 recipes, set a flag `smallCatalog = true` and include this in the AI prompt context (per D-21). The `RecipeSuggestionCard` UI will display based on this, but the AI should also note suggested recipes in its response.

8. **Time check helper** — `function hasTimeLeft() { return Date.now() - startTime < 8500; }` — check before each AI pass. 8500ms leaves 1.5s margin for DB writes.

9. **Pass 1 — Shortlist (Haiku):**
   - System prompt: "You are a household meal planner. Given a recipe catalog and household constraints, return a JSON array of recipe IDs that are good candidates for this week's plan. Consider dietary restrictions (HARD constraint — never include recipes with allergen ingredients), won't-eat lists (HARD — never include), member preferences (ratings), schedule complexity requirements, inventory availability (prefer recipes using ingredients the household has), and budget awareness. Return approximately 30 candidates."
   - User message: JSON containing `{ recipes: [{ id, name, ingredient_names, avg_rating, tags }], constraints: { restrictions, wontEat, schedule, inventory, priorityOrder }, memberCount }`. Sanitize all user-controlled strings (recipe names, food names) by stripping control characters before embedding in the prompt.
   - Model: `claude-haiku-4-5`
   - max_tokens: 1024
   - Parse response: Extract JSON array from response text using regex `text.match(/\[[\s\S]*?\]/)` (same pattern as classify-restrictions). Fallback: use all recipes if parsing fails.

10. **Pass 2 — Assign (model selection per D-06):**
    - Use `claude-haiku-4-5` for households with <=2 members and no dietary restrictions. Use `claude-sonnet-4-5` for households with 3+ members OR active dietary restrictions.
    - System prompt: "You are a household meal planner. Assign recipes to meal plan slots for a 7-day week. Rules: 1) NEVER assign recipes to locked slots. 2) NEVER use recipes containing allergen or won't-eat ingredients. 3) Match recipe complexity to schedule: 'prep' slots get complex recipes, 'quick' slots get simple/fast recipes, 'away' slots get NO assignment. 4) Optimize for the priority order provided (first priority = most important). 5) Prefer recipes that use ingredients the household already has in inventory. 6) Avoid repeating the same recipe more than twice in a week unless the catalog is very small. Return JSON: { slots: [{ day_index: number, slot_name: string, recipe_id: string, rationale: string }], violations: string[], suggestedRecipes: [{ name: string, prepMinutes: number, description: string }] }"
    - User message: JSON with `{ candidates: [...shortlisted IDs with recipe details], slotsToFill: [{ day_index, slot_name, scheduleStatus }], lockedSlots: [...], inventory: [...], nutritionTargets: [...], priorityOrder, smallCatalog }`. Note: slot_name values should use the DB convention. When schedule data uses "Snack" and the plan grid uses "Snacks", normalize to match whichever the meal_plan_slots table actually uses for the existing rows.
    - max_tokens: 4096
    - Parse response: Extract JSON object from response text using regex `text.match(/\{[\s\S]*\}/)`.

11. **Passes 3-5 — Verify and Correct (up to 3 more):**
    - Only run if `hasTimeLeft()` AND previous pass returned `violations.length > 0`
    - Model: `claude-haiku-4-5` (verification is fast)
    - System prompt: same as Pass 2
    - User message: `{ previousAssignment: { slots, violations }, constraints: [same constraint snapshot], instruction: "Fix these violations while preserving as many good assignments as possible." }`
    - max_tokens: 4096
    - Parse and update `bestResult` if the new result has fewer violations

12. **Bulk write slot assignments:**
    - For each slot in the best result's `slots` array: upsert into `meal_plan_slots` on conflict `(plan_id, day_index, slot_name)`. For each slot:
      - Find or create a `meal` row matching the recipe (query `meals` where `household_id` matches and the meal wraps the recipe; if none exists, create one with `name = recipe.name`)
      - Set `meal_id` to the meal's ID
      - Set `generation_rationale` to the slot's `rationale` text
      - Set `is_override = false`
      - Do NOT touch `is_locked` (preserve existing value)

13. **Update job row:**
    - If completed within time: `status = 'done'`, `pass_count = actualPassCount`, `completed_at = now()`
    - If timeout: `status = 'timeout'`, `pass_count = passesCompleted`
    - Store `suggestedRecipes` (if any from D-21) in `constraint_snapshot.suggestedRecipes` field
    - On any error in the AI loop: `status = 'error'`, `error_message = String(err)`

14. **Return response** — The initial response (step 4) already returned `{ success: true, jobId }`. If running synchronously, this is the response after all processing.

**CRITICAL implementation note:** Since this is a Deno edge function and `waitUntil` may not be available, implement the SYNCHRONOUS approach: run the entire flow, then return the response with jobId + final status. The client polls for the job anyway, and the function will have completed by the time the first poll arrives. If the function does support `waitUntil`, the response can be sent early. But do NOT assume `waitUntil` exists.

**Error handling:** Wrap the entire AI loop in try/catch. On error, update the job row to `status: 'error'` with the error message. Always return a valid JSON response (never throw unhandled).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> Edge Function | Untrusted householdId in request body |
| Edge Function -> Anthropic API | User-controlled strings in AI prompts |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Elevation of Privilege | generate-plan request | mitigate | Validate householdId against household_members table for the requesting user's JWT before processing — match classify-restrictions auth pattern |
| T-22-02 | Tampering | AI prompt injection via recipe names | mitigate | Sanitize user-controlled strings (recipe names, food names, won't-eat entries) by stripping control characters before embedding in AI prompts; never place user strings in system prompt |
| T-22-03 | Denial of Service | Repeated generation requests | mitigate | Rate limit: check plan_generations table for household jobs in last 24 hours; reject if >= 10 (per D-07) |
| T-22-04 | Tampering | AI-generated recipe IDs | mitigate | Validate that recipe_ids from AI response exist in the household's recipe catalog before writing to meal_plan_slots; ignore unknown IDs |
| T-22-05 | Information Disclosure | API key exposure | mitigate | ANTHROPIC_API_KEY stored as Supabase Edge Function secret; never returned in response; all AI calls server-side only |
</threat_model>
