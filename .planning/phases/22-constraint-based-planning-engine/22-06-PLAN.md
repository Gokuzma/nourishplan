<objective>
Close UAT Gap B (test 1, truth: "generation completes under time budget"). Currently every recent plan_generations row is stuck at status='timeout' with pass_count=2 because the self-imposed 6-second wall-clock budget expires mid-Pass-2. The edge function mislabels a successful 2-pass run as a timeout failure, giving the UI false failure signals and preventing correction passes 3-5 from ever running.

This plan makes three changes to `supabase/functions/generate-plan/index.ts`:
1. Raise the wall-clock budget from 6000ms to 90000ms (90s) as a named `WALL_CLOCK_BUDGET_MS` constant with a comment referencing the 150s Edge Function platform ceiling. This leaves 60s of headroom for DB writes and platform overhead.
2. Rewrite the status-assignment logic so status='timeout' only fires when Pass 2 (assign) did NOT complete. If Pass 2 completed but correction passes 3-5 were skipped for time, status='partial'. If all active passes completed cleanly, status='done'. If an exception is thrown, status='error' (unchanged).
3. Add 'partial' as a valid value in both the TypeScript union (`src/types/database.ts`) and the Postgres CHECK constraint (new migration `028_plan_generations_partial_status.sql`).

This plan does NOT touch snack fallback logic (Gap A — deferred to plan 22-07 which depends on this one because both edit the same edge function file) and does NOT touch PlanGrid or shimmer rendering (Gap D — plan 22-08, wave 2 sequential after this plan because both edit `src/components/plan/PlanGrid.tsx`).

**D-05 amendment:** This plan amends CONTEXT.md decision D-05 ("10-second wall-clock time budget — return best result if timeout. Edge Function returns partial/best-so-far if time runs out"). The 10s figure was a conservative pre-deployment estimate; the confirmed Edge Function platform ceiling is 150s, and real UAT runs on the test household show Pass 2 alone taking 5-10s, which leaves no budget for the correction passes 3-5 that D-04 requires. The new budget is 90s (150s cap minus 60s headroom for DB writes and platform overhead). D-04's correction loop semantics are unchanged — the loop still runs up to 5 passes max, just with more wall-clock room to actually finish them. The "return partial/best-so-far if time runs out" half of D-05 is preserved exactly: this plan formalises it as the new `'partial'` status value.

**Integration with uncommitted WIP:** The working tree already contains slot-name capitalization fixes in `supabase/functions/generate-plan/index.ts`. Those changes must be preserved. Before executing this plan the executor MUST run `git diff HEAD -- supabase/functions/generate-plan/index.ts` to confirm the capitalization diff is still present; if it is, the executor rebases the edits below on top of the working tree (does not `git stash` or revert).

Purpose: Restore accurate status reporting so the UI shows real success/failure and the correction-pass loop can actually run when violations exist.
Output: A redeployed edge function where typical generation runs on the test household finish in 20-40s with status='done' or 'partial', never 'timeout'.
</objective>
