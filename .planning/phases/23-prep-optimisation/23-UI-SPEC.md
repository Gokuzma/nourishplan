# Phase 23 — UI Design Contract

> Visual and interaction contract for Phase 23: Prep Optimisation.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

Covers three linked surfaces: (1) Batch prep summary modal on the Plan page, (2) full-page Cook Mode route family (`/cook/:mealId`, `/cook/session/:sessionId`, `/cook`), (3) freezer-friendly badge system on recipe cards, plan SlotCards, and the batch prep modal. Plus: recipe editor Steps section, freezer toggle, notification permission prompt, and batch prep stale indicator.

| Property | Value |
|----------|-------|

**No new color tokens introduced.** Phase 23 reuses `--color-primary`, `--color-secondary`, `--color-accent`, `--color-background`, `--color-surface`, `--color-text`, and existing Tailwind utility colors (red-500, amber-500, sky-blue via `#93C5FD`/`#BFDBFE`). Freezer-friendly semantic uses a muted sky-blue tint applied through utility classes only — no new CSS variable.

## Visual Hierarchy

Named focal point per primary screen. The executor must preserve these weightings during implementation — additions to any screen should not displace the stated focal point.

- **Plan page:** `BatchPrepButton` sits adjacent to the existing `GeneratePlanButton` and shares equal visual weight — neither dominates. Both are primary CTAs serving the same user job ("set up this week's cooking"), so they live in the same header row, share the same button height (`py-2.5`), and use complementary chrome (Generate Plan = solid primary; Batch prep = outlined primary-on-secondary).
- **Batch prep modal:** The first (earliest-in-week) `BatchPrepSessionCard` is the focal point — it sits at the top of the scroll body and carries the most urgent "Due soon" chip when applicable. When the stale indicator row is visible it temporarily draws attention (amber pulse above all session cards), but it self-dismisses on recompute and attention returns to the first session card.
- **Cook Mode active step screen:** The active `CookStepCard` is the primary visual anchor — it is the only card with a 2px primary border, `bg-surface` (vs. `bg-secondary` for others), promoted 16px text, and a shadow. When a timer is running on a passive step, the `CookStepTimer` countdown block becomes the secondary anchor during the wait (30px countdown pulling the eye). The fixed footer `CookStepPrimaryAction` is a persistent tertiary — always in view but never competing with the active card chrome.
- **Cook Mode complete state (all steps done):** The "Finish cook session" footer `CookStepPrimaryAction` button is the focal point — no active card remains, so the sage primary footer owns all attention as the one thing left to do.
- **Standalone `/cook` picker:** When the "Resume in progress" section is present (user has an unfinished session), it is the focal point at the top of the scroll body; otherwise the search input above the recipe list is the focal point.
- **Recipe editor Steps section:** The active (currently-being-edited) step row is the focal point — it gets a subtle focus ring on the text input. The "Regenerate from ingredients" affordance in the section header is a secondary anchor, always visible but quieter (`text-xs text-primary underline`, no button chrome).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label (meta, chips, helpers) | 12px (`text-xs`) | 400 / 500 / 600 | 1.4 |
| Body (default) | 14px (`text-sm`) | 400 / 500 / 600 | 1.5 |
| Kitchen-ergonomic body (Cook Mode active step only) | 16px (`text-base`) | 500 (medium) | 1.5 |
| Heading (modal titles, section titles) | 20px (`text-xl`) | 600 / 700 | 1.2 |
| Display (page heading) | 24px (`text-2xl`) | 700 (bold) | 1.2 |
| Timer countdown (Cook Mode only) | 30px (`text-3xl`) | 700 (bold) `tabular-nums` | 1 |

**Weights used:** 400 (regular), 500 (medium), 600 (semibold), 700 (bold).

| Weight | Usage |
|--------|-------|
| 400 regular | Default body text, inactive/completed step text, body copy in banners and prompts |
| 500 medium | Active step text in Cook Mode, freezer badge label, lane owner chip, pill chips, segmented control labels, MultiMealSwitcher pills, Recipe editor Freezer toggle label |
| 600 semibold | Modal headers, section headers, primary buttons, lane header labels, batch prep session header, Cook Mode subtitles, Recipe editor Steps section heading |
| 700 bold | Modal primary titles (batch prep, multi-meal prompt), Cook Mode page display heading, timer countdown |

**Weight palette justification:** Four weights (400/500/600/700) match the existing project weight palette used throughout `src/components/plan/SlotCard.tsx`, `src/components/plan/GeneratePlanButton.tsx`, `src/components/plan/NutritionGapCard.tsx`, and `src/components/grocery/GroceryItemRow.tsx`. Phase 23 components inherit these weights to maintain visual consistency with existing surfaces — consolidating would visibly break the app's established typography.

Notes:
- All text uses `font-sans` class (Nunito Variable)
- Batch prep modal title: `text-xl font-bold text-text` — matches existing modal title pattern in `PlanGrid.tsx` MealPicker (`font-semibold`) but bumped to bold for primary-surface emphasis
- Cook Mode page heading ("Cook Mode" with meal name): `text-2xl font-bold text-primary` for the route label, with meal name as `text-xl font-semibold text-text` below it
- Active step text in Cook Mode: `text-base font-medium text-text` (16px) — step instructions need to be readable at arm's length in the kitchen
- Inactive/completed step text: `text-sm text-text/60 font-sans`
- Timer display: `text-3xl font-bold tabular-nums text-text` — large, legible countdown during passive waits. `tabular-nums` prevents width jitter as digits change
- Freezer badge label text: `text-xs font-medium` (same as ExpiryBadge)
- Recipe editor Steps section heading: `text-sm font-semibold text-text` (matches existing editor section headings)
- Stale indicator text: `text-xs text-text/50 font-sans italic`
- Per-member lane header label: `text-xs font-semibold uppercase tracking-wide text-text/60`
- Notification permission prompt body: `text-sm text-text/80 font-sans`

**Size inventory confirmation:** 12px / 14px / 16px / 20px / 24px / 30px = 6 sizes. The 16px active-step text and 30px timer countdown are stated exceptions to the 3–4 rule with Cook Mode functional justification: 16px is required for kitchen-ergonomic readability at arm's length while cooking; 30px is required for cross-room timer visibility during passive waits. Both sizes are scoped strictly to Cook Mode — the 16px class appears only on the single active-step instruction block, and the 30px class appears only on the countdown timer. All chrome and all non-Cook-Mode surfaces stay within the 12 / 14 / 20 / 24 scale. Both exceptions are functional requirements of the cooking context, not aesthetic choices.

Source: SlotCard.tsx typography patterns, 22-UI-SPEC.md font scale, ExpiryBadge.tsx pill text, NutritionGapCard.tsx headers.

- `bg-accent/20 text-accent` — "Use within N days" fridge hint on batch prep storage suggestions (same tint as ExpiryBadge warning level)

### Freezer-friendly badge color rationale

Sky-blue tint (`bg-sky-100 text-sky-700` light; `bg-sky-900/30 text-sky-400` dark) was chosen for these reasons:
1. Visually distinct from every existing badge semantic (primary=sage, accent=peach, amber=warning, red=danger) — no confusion with allergy/violation badges
2. Cold / frozen mental model association is universal
3. Uses Tailwind's built-in sky palette — no new CSS variable needed
4. Contrast passes WCAG AA in both light and dark modes
5. Matches the existing amber/red ExpiryBadge family (Phase 17) which already uses Tailwind color palettes directly rather than theme tokens for semantic badges

Source: `src/styles/global.css`, SlotCard.tsx badge patterns, ExpiryBadge.tsx color scheme, NutritionGapCard.tsx border patterns, 22-UI-SPEC.md accent reservation list.

## Freezer-friendly badge (visual contract — dedicated)

**Icon:** inline snowflake SVG, 12px × 12px, stroke-based (matches existing inline SVG icon style from LockBadge, ExpiryBadge).

```
<svg width="12" height="12" viewBox="0 0 16 16" fill="none" stroke="currentColor" strokeWidth="1.5" strokeLinecap="round">
  <path d="M8 2v12M2 8h12M3.5 3.5l9 9M12.5 3.5l-9 9" />
</svg>
```

**Pill structure:** `inline-flex items-center gap-1 px-2 py-0.5 rounded-full text-xs font-medium`

**Variants by placement:**

| Placement | Background | Text | Icon color | Label text |
|-----------|-----------|------|------------|------------|
| Recipe card (list) | `bg-sky-100 dark:bg-sky-900/30` | `text-sky-700 dark:text-sky-400` | currentColor | "Freezes well" |
| Recipe detail header | `bg-sky-100 dark:bg-sky-900/30` | `text-sky-700 dark:text-sky-400` | currentColor | "Freezes well · {N} wks" (shelf life if set) |
| Plan SlotCard | icon-only, no background | `text-sky-600 dark:text-sky-400` | currentColor | no label (compact) — 12px icon only, positioned to left of existing right-aligned icon row, before the drag handle's companion icons |
| Batch prep modal session card | `bg-sky-100 dark:bg-sky-900/30` | `text-sky-700 dark:text-sky-400` | currentColor | "Freeze portion" or "Fridge" (storage hint, per D-14) |

**Placement rules:**
- Recipe card: inline next to recipe name, after cost badge if present, before any other tags
- Recipe detail header: next to recipe title in the top metadata row alongside date-created tag
- SlotCard: rendered only on slots where the assigned meal's recipe has `freezer_friendly=true`; placed inside the existing icon row in SlotCard between the member-portion section and the action button cluster; has NO tap behavior (purely informational) — existing icon buttons remain interactive
- Batch prep modal: rendered on every session card, using the storage hint variant. The label varies by AI storage recommendation (fridge vs freezer per D-14) and the tint matches — sky for freezer, accent (peach) for fridge. This is the only placement where the badge can carry non-sky color: an accent-tinted variant appears for fridge-storage sessions.

**Null state:** recipes with `freezer_friendly=null` (unclassified) show **no badge at all** — not a "?" or "Unknown" variant. Lazy backfill (D-10) means users progressively see the badge appear without a confusing intermediate state.

**Explicit-false state:** recipes with `freezer_friendly=false` also show no badge. Only `true` renders the pill.

Source: CONTEXT.md D-09, D-10, D-11, D-12, D-14; ExpiryBadge.tsx pill format; PlanGrid decision to avoid new nav churn.

## Component Inventory

New components required for this phase. All follow existing project patterns.

### BatchPrepButton
- **Placement:** Plan page header row, immediately adjacent to the Phase 22 `GeneratePlanButton`. On mobile these stack vertically; on desktop they sit side-by-side in the existing `flex items-center gap-3 flex-wrap` container.
- **Appearance:** Secondary button visual — `bg-secondary border border-primary/30 text-primary rounded-[--radius-btn] px-4 py-2.5 text-sm font-semibold flex items-center justify-center gap-2`. Full-width on mobile, fixed `w-[180px]` on desktop matching GeneratePlanButton.

- **Disabled tooltip:** "Generate a plan first" — shown when `activePlanSlots.length === 0`
- **aria-label:** "View batch prep summary for this week's plan"

### BatchPrepModal
- **Trigger:** Tapping `BatchPrepButton`
- **Layout:** Bottom-sheet on mobile (`rounded-t-2xl`), centered card on desktop (`rounded-2xl`) — **exact reuse of `MealPicker` modal layout from PlanGrid.tsx** (`fixed inset-0 z-50 flex items-end sm:items-center justify-center`, backdrop `bg-black/40 backdrop-blur-sm`, panel `bg-surface max-h-[85vh] sm:max-w-lg flex flex-col`)
- **Header row:** title "Batch prep for this week" (`text-xl font-bold text-text font-sans`) + total time summary pill ("~90 min total" in `bg-primary/10 text-primary text-xs rounded-full px-2 py-0.5`) + close button (`×` glyph top-right `text-text/40 hover:text-text`). Header border `border-b border-accent/20`.
- **Stale indicator row (conditional):** when `isStale === true` (plan changed within last 30s), renders a pulsing amber row below the header: `bg-amber-500/10 border-b border-amber-500/30 px-4 py-2 text-xs text-amber-700 dark:text-amber-400 font-sans flex items-center gap-2`, copy: "Plan changed — refreshing in {N}s". Includes a small 8px amber dot with `animate-pulse`. Replaced by the fresh summary when the debounce fires.
- **Body:** scrollable (`overflow-y-auto flex-1 p-4 space-y-3`) stack of `BatchPrepSessionCard` components. Empty state (no prep needed this week): centered text "No prep ahead needed — every meal is quick or day-of cook" in `text-sm text-text/50 text-center py-8 font-sans`.
- **Footer row:** single `bg-primary text-white rounded-[--radius-btn] px-4 py-2.5 text-sm font-semibold` button: "Start batch prep cook mode". Tapping this routes to `/cook` with a multi-meal prep session pre-selected for the earliest session card. Border top `border-t border-accent/20`.

### BatchPrepSessionCard
- **Appearance:** `bg-secondary rounded-[--radius-card] p-4 flex flex-col gap-3`
- **Header row:** session label (e.g. "Sunday afternoon — shared rice & beans") in `text-sm font-semibold text-text`, right-aligned prep-time chip (e.g. "~35 min") in `bg-primary/10 text-primary text-xs rounded-full px-2 py-0.5`. If the session is due within 24 hours, an additional `bg-amber-500/10 text-amber-700 text-xs rounded-full px-2 py-0.5` chip labeled "Due soon".
- **Recipes list row:** 2–4 small recipe pills stacked vertically. Each: 32px recipe name prefix, total prep minutes in `text-xs text-text/50`, freezer-friendly badge (sky or accent per storage recommendation) if `freezer_friendly=true`. Example: "Brown rice · 25 min [Freeze portion]" / "Black beans · 20 min [Fridge 3 days]".
- **Shared ingredients callout (conditional):** when the AI detects shared ingredients, a highlighted row below the recipe list: `bg-primary/10 text-primary text-xs rounded-[--radius-btn] px-3 py-2 font-sans` with a small info-dot icon prefix. Copy format: "2 cups rice cooked once for 3 meals" (per CONTEXT.md D-14 verbatim example).
- **Equipment callout (conditional):** separate small inline text row below shared ingredients when AI flags an equipment note: `text-xs text-text/50 font-sans` prefixed with a 12px stovetop icon. Example: "Uses the large pot for all three". Optional — only rendered if AI provides equipment metadata.
- **Storage hint row:** single row at the card bottom listing per-recipe storage recommendations via the freezer-friendly badge variant. Example: "Rice → [Freeze portion], Beans → [Fridge 3 days], Roasted peppers → [Fridge 5 days]". Inline wrapping flex `flex flex-wrap gap-1.5 items-center`.
- **"Cook this session" action:** tap-anywhere card button opens Cook Mode at `/cook/session/{sessionId}` with the multi-meal prompt shown on entry (per D-21 prep-slot-with-multiple-recipes flow). Keyboard accessible as a single `<button>` wrapping the card content (role="button", existing pattern from GroceryItemRow.tsx).

### CookModeShell (page-level layout for `/cook/:mealId` and `/cook/session/:sessionId`)
- **Route container:** `min-h-screen flex flex-col bg-background` — full viewport, escapes the AppShell layout route by using a top-level route outside `<Route element={<AppShell />}>`
- **Top bar:** fixed `sticky top-0 z-40 bg-surface border-b border-accent/20 px-4 py-3 flex items-center justify-between`
  - Left: back button (16px ← chevron, `text-text/60`), `aria-label="Exit cook mode"` — on tap shows confirmation if any step is checked or any timer running ("Exit cook mode? Progress will be saved.") otherwise navigates back
  - Center: meal name (`text-sm font-semibold text-text truncate`) above tiny subtitle (`text-xs text-text/50`) showing "Day X · Slot Y" when launched from a plan slot, or "Standalone" otherwise
  - Right: `MultiMealSwitcher` (below) when multiple concurrent sessions exist, else a menu icon (`⋮`) that opens overflow
- **Progress bar:** directly below top bar. `h-1.5 bg-primary/20` track with `bg-primary transition-all duration-300` fill, width proportional to `completedSteps / totalSteps`. Screen reader text: `role="progressbar" aria-valuenow={pct} aria-valuetext="{N} of {M} steps complete"`
- **Body:** `flex-1 overflow-y-auto px-4 py-4 space-y-3 pb-32` — padding bottom clears the fixed footer
- **Footer:** fixed `fixed bottom-0 inset-x-0 z-40 bg-surface border-t border-accent/20 px-4 py-3` containing the active-step primary action (see `CookStepPrimaryAction`)

  - Active/passive indicator chip: active-hands-on = `bg-accent/10 text-accent text-xs rounded-full px-2 py-0.5` labeled "Hands-on"; passive-waiting = `bg-primary/10 text-primary text-xs rounded-full px-2 py-0.5` labeled "Passive wait"

  - Equipment line (conditional): `text-xs text-text/50 font-sans` with stovetop/oven/pot icon prefix
  - Lane owner chip (conditional): "{Member name} is doing this" in `text-xs text-primary bg-primary/10 rounded-full px-2 py-0.5`
  - Timer block (conditional): visible when step duration >0 and timer has been started — see `CookStepTimer` below

### CookStepTimer
- **Structure:** centered block inside the active CookStepCard, `flex flex-col items-center gap-2 py-3`
- **Countdown display:** `text-3xl font-bold tabular-nums text-text` format "M:SS" (e.g., "12:34")
- **Progress ring behind countdown (optional):** 64px circular SVG ring, `stroke="var(--color-primary)"` with `strokeDasharray` animated — reuses the existing `ProgressRing` component from `src/components/plan/ProgressRing.tsx` with size 64, strokeWidth 4
- **State colors:**
  - Running (>60s remaining): primary color countdown text
  - Warning (<60s remaining): text color shifts to `text-amber-500`, ring color to `amber-500`, gentle `animate-pulse` applied to the entire block
  - Complete (0s): text replaced by inline check mark icon + "Done" label, color `text-primary`; triggers a brief `animate-bounce` for 2 seconds
- **Controls row:** directly below countdown, `flex gap-2`
  - Primary: "Start" (before timer runs) / "Pause" (while running) / "Reset" (when paused or complete) — `bg-primary text-white rounded-[--radius-btn] px-4 py-2 text-sm font-semibold`
  - Secondary: "Skip" — `text-text/60 hover:text-text text-xs underline` (no button chrome)
- **Notification permission prompt trigger:** tapping "Start" on the first passive-wait step of a user's first-ever cook session triggers `NotificationPermissionPrompt` inline above the CookStepCard (see below)

### CookStepPrimaryAction (fixed footer button)
- **Appearance:** full-width `bg-primary text-white rounded-[--radius-btn] h-12 text-sm font-semibold flex items-center justify-center gap-2`
- **Label varies by active step state:**
  - Active step, timer not started, no duration: "Mark complete" (with 16px check icon prefix)
  - Active step, timer not started, duration >0: "Start timer" (with 16px play icon prefix)
  - Active step, timer running: "Mark complete now" (with 16px check icon prefix) — allows user to advance before timer finishes
  - Active step, timer done: "Mark complete" (with 16px check icon prefix), button gets a subtle `animate-pulse` hint
  - Last step active: "Finish cook session" (with 16px check icon prefix)
  - No active step (all complete): "Exit cook mode" with destructive-tinted primary (`bg-primary` still — not red, because exit isn't destructive)
- **After tap:** advances to next step with 300ms fade-out/fade-in transition on the CookStepCard; smooth scroll to the new active step

### MemberLaneHeader
- **Placement:** rendered as a sticky subheader within CookModeShell body above any step owned by the given member. Only appears when the cook session has >1 active lane (per CONTEXT.md D-23 per-member lanes).
- **Appearance:** `sticky top-[calc(top-bar-height+progress-bar-height)] bg-background/95 backdrop-blur-sm px-4 py-2 z-30 flex items-center gap-2 border-b border-accent/20`
- **Content:**
  - 24px member avatar circle (existing avatar pattern from Household members)
  - Member name `text-xs font-semibold uppercase tracking-wide text-text/60`
  - Step count progress `text-xs text-text/50 font-sans`, e.g. "2 of 4 done"
  - Optional "Swap owner" button: `text-xs text-primary underline`, opens a tiny member picker inline — letting the household adjust lane ownership on the fly

### MultiMealSwitcher
- **Trigger:** Right-side control in CookModeShell top bar, visible only when `concurrentSessions.length > 1`
- **Collapsed state:** horizontal pill strip `flex gap-1 max-w-[40vw] overflow-x-auto scrollbar-none`. Each pill: `px-2 py-1 rounded-full text-xs font-medium` with a 6px status dot prefix (primary=active, accent/60=paused, text/30=done). Example: "• Breakfast", "• Lunch", "• Dinner"
- **Active pill:** `bg-primary text-white`
- **Inactive pills:** `bg-secondary text-text/60`
- **Tap behavior:** switches the CookModeShell body to the chosen session's state; route changes to `/cook/session/{sessionId}`; the current session's state is already persisted server-side per D-22 so no save prompt is needed
- **Accessibility:** pills are `<button>` elements inside a `role="tablist"` container, each pill `role="tab" aria-selected="true|false"`

### MultiMealPromptOverlay

## Interaction Contracts

### Batch Prep Modal Open + Auto-Refresh Flow (PREP-01, D-13, D-17)

1. User taps `BatchPrepButton` on Plan page
2. Modal opens with backdrop fade (200ms)
3. If cached batch prep summary exists for current plan: renders immediately
4. If no cached summary: body shows skeleton state (3 `SlotShimmer`-style session cards for 800ms minimum)
5. TanStack Query fetches `computeBatchPrep` edge function result; response hydrates session cards
6. When user drags any slot on the Plan page while the modal is open:
   - `isStale` becomes true immediately
   - Stale indicator row (`BatchPrepStaleIndicator`) appears with "Plan changed — refreshing in 30s" copy
   - Countdown text updates every second
   - If user makes another change, the 30s timer resets (debounce per D-17/D-28)
   - When countdown reaches 0: stale row is replaced by the refreshed summary; brief 300ms fade transition
7. Close modal: tap backdrop, tap × button, or press Escape
8. State persists across reopens until plan changes

### Cook Mode Entry from SlotCard (PREP-02, D-18, D-20a, D-21)

1. User taps the new "Cook" icon button on an occupied SlotCard
2. App navigates to `/cook/{meal.id}?slotId={slot.id}`
3. CookModeShell mounts outside AppShell layout (full-page)
4. **Flow branches on slot.schedule_status** (per CONTEXT.md D-21):
   - **consume:** Fetches reheat sequence via `generate-reheat-sequence` edge function; renders `ReheatSequenceCard` only
   - **prep with >1 recipe in the slot's batch:** Shows `MultiMealPromptOverlay` before any steps render; user chooses combined vs per-recipe
   - **prep with 1 recipe, quick, day-of scratch:** Full Cook Mode — fetches/loads recipe.instructions, renders step list, starts at step 0 active
5. Cook session row is upserted into `cook_sessions` table at mount; `session_id` stored in component state
6. Supabase Realtime subscription starts on `cook_sessions.id=eq.{sessionId}` for collaborative sync

### Active Step Advance Flow (D-19)

1. CookStepCard for step N is in active state
2. **For hands-on steps (is_active=true) with no duration:**
   - `CookStepPrimaryAction` label: "Mark complete"
   - User taps: session step state updates (`cook_sessions.steps[N].completedAt = now`), step card transitions to completed state, step N+1 becomes active, auto-scroll to it, progress bar fill updates
3. **For steps with duration:**
   - `CookStepPrimaryAction` label: "Start timer"
   - User taps: `CookStepTimer` appears within the active card; countdown starts
   - If this is user's first passive-wait timer, `NotificationPermissionPrompt` renders above the active card before timer starts
   - Countdown runs via `requestAnimationFrame` loop with 1s update interval for text
   - At 60s remaining: amber warning state engages
   - At 0s: timer block shows "Done" with bounce animation; service worker fires local notification if permission granted; primary action button label becomes "Mark complete" with pulse animation
   - User taps "Mark complete": advances to step N+1
4. **Skip button behavior:** tapping "Skip" during a running timer: pauses timer, shows confirmation inline "Skip the rest of this wait?" — confirming marks step complete and advances

### Collaborative Cook with Per-Member Lanes (D-22, D-23, D-24)

1. Two household members open the same `/cook/{meal.id}?slotId={slot.id}` URL on their own devices
2. Both devices upsert into the same `cook_sessions` row (by slot + meal composite key)
3. AI lane assignment is loaded with the session — step 1 owned by Member A, step 2 owned by Member B, etc.
4. On each device:
   - Steps owned by the viewing member display with primary-tinted lane indicator chip
   - Steps owned by the other member display with muted lane indicator
   - `MemberLaneHeader` sticky subheader appears above each owner's lane block
5. When Member A checks off step 1:
   - Immediate optimistic update on Member A's device (step completed, advance)
   - Supabase Realtime pushes the update to Member B's device
   - On Member B's device, step 1 transitions to completed state with a subtle fade-in highlight (`bg-primary/20` flash for 600ms) indicating cross-device update
6. If a step has no owner (AI didn't assign): any member can check it off; first-write-wins
7. Owner swap: tapping "Swap owner" on a MemberLaneHeader opens a small inline picker; change syncs via realtime

### Multi-Meal Concurrent Sessions (D-26)

1. User opens Cook Mode for Breakfast → session B created
2. User taps back to Plan page, opens Cook Mode for Lunch → session L created
3. Both sessions persist state in `cook_sessions`
4. Cook Mode top bar now shows `MultiMealSwitcher` with pills "• Breakfast" and "• Lunch"
5. Tapping a pill switches the displayed session
6. Each session's progress, timers, and step states are isolated

### Plan Auto-Reassignment on Missing Prep Session (D-16)

1. User opens batch prep modal
2. Edge function detects that Wednesday's "consume" slot for Lentil Soup has no matching prep session earlier in the week
3. Edge function auto-writes a reassignment — e.g., moves the recipe to Monday's prep slot
4. Response returns both the new session summary AND the slot reassignment
5. Modal renders with the updated session card
6. Back on the Plan page (visible behind the modal overlay backdrop): the moved slot briefly highlights with a `bg-primary/20` pulse for 800ms indicating it was auto-moved. A small toast at the top of the Plan page says "Moved Lentil Soup prep to Monday for shared cooking time" with an "Undo" button

### Recipe Step Regeneration After Ingredient Edit (D-04, D-27)

1. User edits an ingredient in an existing recipe that has AI-generated + user-edited steps
2. On save, the RecipeEditorStepsSection state chip switches to "Regenerating..."
3. Edge function receives the previous user-edited steps as merge-intent context
4. AI returns revised steps + (optionally) a list of "uncertain" user additions
5. The section re-renders with the new steps
6. If any uncertain additions: inline amber callout(s) appear above the relevant steps with "Keep/Remove" quick actions per callout

### Freezer Badge Placement in Context (D-11)

- Recipe list card: rendered in the existing tag row, after cost badge
- Recipe detail page: rendered inline next to recipe title in metadata row
- Plan SlotCard: rendered as icon-only to the left of the action button cluster
- Batch prep modal session card: rendered per-recipe in the storage hint row, with fridge-variant or freezer-variant tint based on AI recommendation

### Notification Permission First-Run (D-25)

1. User enters first cook session with passive-wait step
2. Taps "Start timer" on the passive step
3. `NotificationPermissionPrompt` inline banner appears above the active CookStepCard
4. User taps "Allow notifications" → `Notification.requestPermission()` fires native browser prompt
5. On grant: banner disappears, timer continues normally
6. On deny: banner replaces itself with the "Notifications blocked" state; user can dismiss
7. On subsequent sessions after grant: service worker handles notifications silently; banner never shows
8. On subsequent sessions after deny: banner re-shows after 7-day cooldown

## Mobile / PWA / Offline Considerations

- **Cook Mode is the primary kitchen-hands-busy use case.** Active step text size is promoted to 16px (`text-base`) per Typography; primary action button height is 48px; timer countdown is 30px for across-the-kitchen legibility.
- **Cook Mode offline readiness:** Recipe instructions are cached on the recipe record (`recipes.instructions` JSONB per D-01), so once a recipe has been viewed online, Cook Mode can run fully offline for that recipe. Timer countdown logic is pure-client. Step check-offs queue locally until online (existing TanStack Query mutation queue pattern from Phase 4 offline sync).
- **Supabase Realtime gracefully degrades when offline:** The cook_sessions subscription drops; on reconnect, the full session state is re-fetched and merged with any local optimistic check-offs.
- **Service worker notifications (D-25)** only fire when the browser grants permission. When denied, passive step timers still count down visually but no notification fires.
- **Batch prep modal offline:** if there's no cached batch prep result, the modal shows "Connect to view batch prep" message with a retry button. Cached results (TanStack Query persistence) remain viewable offline.
- **MultiMealSwitcher works offline:** switching between concurrent local sessions is pure-client; cross-device sync pauses until reconnect.
- **Installed PWA consideration:** the notification permission prompt flow explicitly uses `Notification.requestPermission()` — service worker registration is already handled by `vite-plugin-pwa` so no additional registration code is needed at Cook Mode level. Service worker `showNotification()` fires local notifications with tag deduplication (per-session tag so rapid timer completions don't stack).
