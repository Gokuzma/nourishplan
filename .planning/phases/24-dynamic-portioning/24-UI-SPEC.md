Warning amber (`text-amber-500`) is already used in the codebase for NutritionGapCard — reuse it for the sum-not-100% validation state (sliders do not sum to 100).

## Component Inventory

### New Components

#### `RecipeMixPanel`

Location: `src/components/plan/RecipeMixPanel.tsx`

A collapsible panel placed on the Plan page directly below `PriorityOrderPanel`. Follows the identical expand/collapse pattern:

- Collapsed state: full-width `bg-secondary rounded-[--radius-card]` button with label "Recipe mix" and a chevron icon.
- Expanded state: panel body reveals three labeled slider rows (Favorites / Liked / Novel) with percentage values.
- Chevron rotates 180° on expand (`transition-transform`), matching `PriorityOrderPanel`.

**Slider row anatomy (per slider):**
- Row wrapper: `flex items-center gap-3 px-3 py-2` with min-height 44px
- Label: `text-sm text-text font-sans flex-1`
- `<input type="range" min="0" max="100" step="5">` — native range input, styled via Tailwind (`accent-[--color-accent]` for cross-browser thumb color)
- Value readout: `text-xs text-text/60 w-8 text-right` (e.g. "50%")

**Validation row (below the three sliders):**
- Shows running total of all three values
- When total = 100: `text-xs text-primary font-sans` → "Total: 100% ✓"
- When total ≠ 100: `text-xs text-amber-500 font-sans` → "Total: {N}% — must equal 100"
- No form submission blocked by this; validation is advisory only. Sliders auto-rebalance: adjusting one slider proportionally redistributes the remainder across the other two.

**Persistence:** localStorage key `plan-recipe-mix-{householdId}` (matches `PriorityOrderPanel` pattern). Default values: Favorites 50%, Liked 30%, Novel 20%.

**Saved confirmation:** Same pattern as `PriorityOrderPanel` — "Saved" text in `text-xs text-primary` fades in for 1.5 seconds after any change.

#### No other new components

All other phase 24 changes are backend (edge function queries, AI prompt) or text-only modifications to existing `AIRationaleTooltip` content. No new pages, no new routes, no new modals.

### Modified Components

#### `AIRationaleTooltip` content (text only — no markup changes)

The existing tooltip markup and behaviour are unchanged. Only the text content fed to it changes. The executor must ensure the generate-plan edge function produces rationale strings that include preference tier context.

Required rationale format patterns (for the executor's reference):
- Favorite tier: `"Favorite — avg {N} stars across {N} cooks"`
- Liked tier: `"Liked — last cooked {N} weeks ago"`
- Novel tier: `"Novel — similar ingredients to your top-rated {Recipe Name}"`
- Fallback (no history): existing rationale text, unchanged

Tooltip visual contract (already implemented, do not change):
- `absolute z-50 mt-1 bg-surface border border-accent/30 rounded-[--radius-card] shadow-md px-3 py-2 text-xs text-text/70 font-sans max-w-[240px]`

## Interaction Contract

### Recipe Mix Sliders

**Auto-rebalance rule:** When the user drags slider A, compute remaining = 100 - A. Distribute remaining proportionally across B and C based on their current ratio. If both B and C are 0, split remaining 50/50. Clamp all values to [0, 100] at 5% steps. This keeps the total at 100% at all times, eliminating the need for an error state in practice — the validation row is a fallback for direct value edits.

**Slider interaction states:**
- Default: thumb at current value, track filled from 0 to thumb in `bg-accent/40`
- Focus: native browser `:focus-visible` ring (do not suppress)
- Dragging: thumb scales to 1.2x via CSS `transform` on `:active` (`active:scale-110` on the input)

**Accessibility:**
- Each `<input type="range">` has an `aria-label` (e.g. `"Favorites percentage"`)
- Each `<input>` is associated with a visible `<label>` via `htmlFor`/`id`
- The validation row has `role="status"` so screen readers announce total changes

**Touch target:** The row wrapper (`flex items-center gap-3 px-3 py-2`) with `min-h-[44px]` ensures the slider meets WCAG 2.5.5.
