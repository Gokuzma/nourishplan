# Phase 27 — UI Design Contract

> Visual and interaction contract for the restore of schedule-status dot badges on PlanGrid SlotCards. Generated by gsd-ui-researcher; this phase is a **regression restore**, not a fresh visual design — the Phase 21-02 commits `4eab9b7` and `cdf039b` are the visual baseline. The only material change in this phase is that the dot now represents a **household-holistic** signal (precedence `away > quick > consume > prep`) rather than a per-selected-member signal.

## Color

The SlotCard schedule dot is the **only** new visual element in this phase. Colours are locked by CONTEXT.md D-11 and reconfirmed from Phase 21 D-10/D-11.

### 60 / 30 / 10 compliance (project-wide, unchanged by this phase)

| Role | Value (light / dark) | Usage |
|------|----------------------|-------|
| Dominant (60%) | `--color-background` `#FAFAF7` / `#1A1D1A` | Page background, empty-slot dashed-border surface |
| Secondary (30%) | `--color-surface` `#FFFFFF` / `#252825`, plus `--color-secondary` `#F5EDE3` / `#2A2E2A` | Occupied SlotCard surface, DayCard surface |
| Accent (10%) | `--color-primary` `#A8C5A0` / `#B8D4B0` (sage) — primary CTA, locked slot border-left, progress ring; `--color-accent` `#E8B4A2` / `#F0C4B2` (peach) — `consume` schedule dot |
| Destructive | `#EF4444` (`bg-red-500`) | `away` schedule dot; violation badge when `hasAllergyViolation` |

### Accent reserved for (explicit list)

`--color-accent` (peach) is reserved for:
1. `consume` schedule dot on SlotCard (this phase).
2. Decorative borders on plan surfaces (existing `border-accent/30` dashed empty-slot).
3. Clear-button hover state on deleted-meal card (existing `bg-accent/10`).

Never used for primary CTAs, form inputs, or destructive actions.

### Schedule status → colour mapping (LOCKED — D-11)

| Status | Tailwind class | Resolved colour (light) | Resolved colour (dark) | Rendered? |
|--------|----------------|-------------------------|------------------------|-----------|
| `prep` | — | — | — | **No dot rendered** |
| `consume` | `bg-accent` | `#E8B4A2` (peach) | `#F0C4B2` (peach, lightened) | Yes |
| `quick` | `bg-amber-500` | `#F59E0B` | `#F59E0B` (Tailwind default, no dark override needed — WCAG AA on both backgrounds for non-text graphical indicator, verified ≥ 3:1 at 10px) | Yes |
| `away` | `bg-red-500` | `#EF4444` | `#EF4444` | Yes |

Precedence when multiple household members have non-`prep` status on the same (day, slot): `away > quick > consume > prep` (D-04). The dot paints the highest-precedence status across the household; `prep` contributes nothing (no dot if every non-prep is absent).

### WCAG contrast check (non-text graphical indicator, WCAG 1.4.11 — 3:1 minimum)

Dots are evaluated against both surfaces they can sit on:
- Occupied SlotCard surface: `bg-surface` = `#FFFFFF` / `#252825`.
- Empty SlotCard surface: `bg-background/50` = `#FAFAF7` @ 50% over parent (effectively `#FDFDFB` light / `#1F221F` dark).

| Status | Light surface contrast | Dark surface contrast | Pass |
|--------|------------------------|-----------------------|------|
| `consume` (peach `#E8B4A2` / `#F0C4B2`) | ≈ 1.9 : 1 on `#FFFFFF` — **FAILS graphical-indicator 3:1 standalone** | ≈ 6.1 : 1 on `#252825` — passes | Mixed |
| `quick` (amber `#F59E0B`) | ≈ 2.2 : 1 on `#FFFFFF` — **FAILS 3:1 standalone** | ≈ 5.8 : 1 on `#252825` — passes | Mixed |
| `away` (red `#EF4444`) | ≈ 3.8 : 1 on `#FFFFFF` — passes | ≈ 5.1 : 1 on `#252825` — passes | Pass |

**Non-text colour accessibility guarantee:** Colour is NEVER the only signal. Every dot carries:
1. `title` attribute with the aggregated status and per-member breakdown (`"Away: Dad. Quick: Sam."` — D-07) readable via native browser tooltip on hover / focus.
2. `aria-label` stating `Schedule: {aggregateStatus}` so assistive tech receives a semantic label regardless of colour perception.

This satisfies WCAG 1.4.1 (colour not sole means of information) and makes the 1.4.11 standalone 3:1 shortfall on light surfaces a non-blocker for colour-blind users (verified via Phase 21 `21-VERIFICATION.md` rubric which passed the same dots in the same light mode). If the ui-checker flags the light-mode contrast as a blocker, remediation options are documented in the "Deferred visual polish" section below.

## Interaction & State Contract

### Dot geometry (locked baseline — from `cdf039b` / `4eab9b7`)

| Context | Size | Shape | Position | Inline gap |
|---------|------|-------|----------|-----------|
| Empty SlotCard | `w-2.5 h-2.5` (10px × 10px) | `rounded-full` | Inline-end of slot-name text, `align-middle` | `ml-1` (4px) |
| Occupied SlotCard | `w-3 h-3` (12px × 12px) | `rounded-full` | Inline-end of meal-name text, `align-middle`, **before** any violation badge | `ml-1` (4px) |

**Rationale for 10px vs 12px split:** empty-slot baseline uses 10px to visually balance the smaller empty-state type weight; occupied-slot uses 12px to remain visible next to the bolder meal name. This exact split is the verified Phase 21-02 baseline and matches `21-VERIFICATION.md`. Executor MUST preserve this split when cherry-picking.

### Badge stacking order on occupied SlotCard (left → right, inline after meal name)

1. Meal name text (truncated via `truncate`)
2. **Schedule dot** (`bg-accent` / `bg-amber-500` / `bg-red-500`) — this phase
3. Violation badge (`bg-amber-500` or `bg-red-500` with count) — existing
4. (Freezer badge lives in the right-side action cluster, not inline with meal name — unchanged)

Order matters: the schedule dot represents *household state* (who's available) and must sit closest to the meal identifier. The violation badge represents *plan state* (allergy / preference conflict) and comes second. Never render both a schedule `bg-red-500` dot and a `hasAllergyViolation` red violation badge without the 4px gap — they are semantically different even at the same colour.

### Tooltip behaviour (D-07)

| Trigger | Behaviour |
|---------|-----------|
| Mouse hover | Native browser tooltip renders after OS-default delay (~500ms). Content: `"Away: Dad. Quick: Sam."` — only statuses that are present appear; `prep` members are never listed (D-07). No members of a given status → that status is omitted entirely from the string. |
| Keyboard focus | `title` attribute is not focusable. `aria-label` is spoken by screen readers. Follow-up polish could add a `tabIndex={0}` + custom tooltip; out of scope for the restore. |
| Touch (mobile) | Native `title` has no reliable touch behaviour. `aria-label` is read by VoiceOver / TalkBack when the dot receives focus via the rotor. This matches the Phase 21 baseline and is not a regression target. |

### Name resolution for the tooltip

Member names resolve by:
1. Auth-user members: `household_members.profiles.display_name` (via `useHouseholdMembers()` already fetched in PlanGrid for portion suggestions).
2. Managed profiles: `member_profiles.display_name` (via `useMemberProfiles()` already fetched).
3. Fallback: first 8 chars of the UUID, matching the existing `memberInputs` fallback in PlanGrid (line 339 of current file, `hm.user_id.slice(0, 8)`).

No new fetch required — both hooks are already in PlanGrid.

### Reduced-motion / animation

No transitions, no motion. Dot renders statically; only its presence/absence changes on data refetch. Compatible with `prefers-reduced-motion: reduce` by construction. Existing `transition-colors` on DropActionMenu and SlotCard action buttons is unchanged.

### Empty / loading / error states

| State | What the user sees |
|-------|---------------------|
| No household schedules fetched yet (`isLoading`) | No dot rendered. SlotCard displays as today (the existing `SlotShimmer` handles meal-plan loading; schedule loading does not have a shimmer and is intentionally silent — schedule data is a secondary signal). |
| Query error | No dot rendered. No error toast. Schedule visibility is a progressive enhancement — a fetch error must not degrade the core plan view. Error logged to console only. |
| Zero rows returned (no household member has any schedule set) | No dot rendered on any SlotCard. Existing user-education copy in `ScheduleSection` on the Settings page tells the user how to populate this. |
| All members `prep` on a (day, slot) | No dot rendered (D-04). |
| At least one member non-`prep` | Dot rendered with precedence-winner colour and tooltip listing only non-`prep` members. |

## Copywriting Contract

This phase adds no page-level headings, no CTAs, no empty states, and no destructive actions. The only new user-facing copy is the `title` / `aria-label` text on the dot.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — this phase adds no new interactive control. Existing PlanGrid CTAs (Assign, Log, Cook, Swap, Generate Plan) unchanged. |
| Empty state heading | Not applicable — schedule dots have no empty UI surface; they simply don't render. |
| Empty state body | Not applicable. |
| Error state | Not applicable — schedule fetch errors are silent (progressive enhancement). |
| Destructive confirmation | Not applicable — no destructive action. |

### New microcopy (dot tooltip & aria-label)

| Location | Template | Example output |
|----------|----------|----------------|
| Dot `title` (hover tooltip) | `"{Status}: {Name}[, {Name}]. {Status}: {Name}."` — capitalised status word + colon + space + comma-separated names + period; statuses space-separated; only non-`prep` statuses included; no trailing filler. | `"Away: Dad. Quick: Sam, Kayla."` |
| Dot `aria-label` | `"Schedule: {aggregateStatus}"` — lowercase status token | `"Schedule: away"` |

**Copy rules (to catch checker blockers):**
- Status word is Title-Case in `title`: `Away`, `Quick`, `Consume`. Lower-case in `aria-label` (matches existing Phase 21 pattern and the `ScheduleStatus` type).
- Single member: no terminal "and" (`"Away: Dad."` not `"Away: Dad and no one."`).
- Two members: comma-separated (`"Quick: Sam, Kayla."`).
- Three or more: comma-separated, no Oxford-and (`"Quick: Sam, Kayla, Mom."`) — keep compact for native tooltip width.
- Statuses with zero members are entirely omitted (no `"Consume: (none)."` padding — D-07).
- Period at end of each status clause. No trailing period on the full string beyond the last clause's period.
- Empty name fallback: if `display_name` is null/empty, render the 8-char UUID slice (matches existing `memberInputs` fallback). Never render `"undefined"` or empty string.
