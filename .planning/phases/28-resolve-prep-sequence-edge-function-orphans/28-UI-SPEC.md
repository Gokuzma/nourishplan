### WCAG contrast check (graphical-indicator + body text on overlay panel)

- Spinner stroke `--color-primary` `#A8C5A0` on `bg-surface` `#FFFFFF`: ≈ 1.9 : 1 — **fails 3:1 standalone** for graphical indicators (WCAG 1.4.11). Mitigation: spinner motion (continuous rotation) is itself a perceptible affordance independent of colour, and the `aria-busy` + `aria-label` semantic announcement covers users who cannot perceive motion. This is consistent with how `GeneratePlanButton.tsx:34-48` uses the same colour pairing on a `bg-primary` button (white spinner on sage).
- Headline text `--color-text` `#3D3D3D` on `bg-surface` `#FFFFFF`: ≈ 11.5 : 1 — passes WCAG AAA.
- Sub-line text `text-text/60` (`#3D3D3D` at 60%) on `bg-surface` `#FFFFFF`: ≈ 6.5 : 1 — passes WCAG AA.
- Dark mode: spinner `#B8D4B0` on `#252825` ≈ 6.8 : 1 — passes 3:1.

**Non-text colour accessibility guarantee:** the overlay's loading semantics are conveyed through three independent signals: (1) continuous spinner rotation, (2) text headline "Planning your cook session…", and (3) `aria-busy="true"` + `aria-live="polite"` on the panel root. Colour is never the sole signal.

### Combined cook flow pending state (CHANGED branch)

`CookModePage.tsx:393-404` currently renders `MultiMealPromptOverlay`. Phase 28 wraps the `onCombined` / `onPerRecipe` handlers so they:
1. Fire `useGenerateCookSequence.mutateAsync(...)` (mutation auto-renders the loading overlay via the `isPending` flag).
2. On success: store the AI sequence on the cook session and transition `setFlowMode('cook')`.
3. On error: silently build a per-recipe concatenation locally (D-05) and transition `setFlowMode('cook')`.

**State machine for the multi-meal-prompt branch:**

| State | What renders |
|-------|--------------|
| Idle (user has not yet clicked) | `MultiMealPromptOverlay` (unchanged from current behaviour). |
| Pending (`useGenerateCookSequence.isPending === true`) | `CookSequenceLoadingOverlay` rendered above (or in place of) the prompt overlay. The prompt's "Back" button is unreachable during pending — this is intentional (CONTEXT.md D-04 disables backdrop dismissal). |
| Success | `CookModeShell` for the new combined cook session (`flowMode === 'cook'`). The loading overlay unmounts cleanly on the same render as the shell mount. |
| Error / fallback (D-05) | `CookModeShell` for a per-recipe-concatenated cook session (`flowMode === 'cook'`). **No banner, no toast, no error indicator.** The session step ordering is naive concatenation (each recipe's steps in original order, no interleaving, no member ownership). |

### Failure-state UX decision (LOCKED — resolves CONTEXT.md "Claude's Discretion" items 2 + 4)

**Decision: silent fall-through for BOTH cook-sequence and reheat-sequence failures. No toast, no banner, no inline notice.**

Rationale:
1. **Consistency with Phase 25 D-10 graceful-degradation pattern** — Phase 25 chose silent fall-through for URL-fetch failures (the user types/pastes recipe text into the same modal with no error message about the URL fetch). The user-facing principle is identical: AI is enhancement, never blocker; the failure mode is invisible because the fallback experience is fully functional.
2. **The fallback experiences are fully functional, not degraded.** A per-recipe concatenated cook session is exactly the pre-Phase-28 default behaviour for multi-recipe sessions. A 3-step microwave reheat is exactly the existing hardcoded `reheat-1/2/3` array. Neither is broken; neither lies. The only loss is "AI didn't optimise this" — an absence of enhancement, not a presence of breakage.
3. **No telemetry phase yet (D-08).** Surfacing a banner like "AI optimization unavailable" without a corresponding "what to do about it" path or a metric to measure recurrence rate just adds noise. When the cost/usage telemetry phase ships (deferred to v2.1+), error-rate observability lands then; banners can be added in a follow-up if real users complain.
4. **Failure is logged to console for developer diagnosis.** Both mutations have `onError` handlers that `console.error('[Phase 28] generate-cook-sequence failed', err)` (or analogous) so devs investigating user-reported issues can find the failure in browser DevTools without it intruding on the user's flow.

**What MUST NOT appear on failure (anti-regression):**
- No `<div role="alert">` element
- No toast / snackbar / `react-hot-toast` call
- No inline `<p className="text-red-...">` error text
- No `aria-live="assertive"` announcement
- No retry button (the user can simply exit and re-enter cook mode if they want a fresh try)

**What SHOULD appear on failure:**
- The fully-functional fallback cook step list / reheat step list
- A `console.error(...)` line in DevTools with the failure details

This decision is reversible in a future polish phase if user feedback indicates the silent fall-through is too opaque. Until then, it is the locked Phase 28 contract.

## Anti-Regression Contract

This section exists because Phase 28 is closing v2.0 audit WARN-01 — two edge functions sat orphaned for an entire milestone because nothing in the codebase asserted they were wired in. The ui-checker, ui-auditor, and any future executor must enforce the following grep guards to prevent the wire-in from being silently undone (matches `27-UI-SPEC.md` discipline; extends L-020 / L-027 to AI-edge-function consumers):

1. **`CookModePage.tsx` MUST import and call `useGenerateCookSequence` AT LEAST ONCE.** Grep assertion: `grep -c "useGenerateCookSequence" src/pages/CookModePage.tsx` returns ≥ 2 (one import line + at least one call site). If this returns < 2, the wire-in has been truncated.
2. **`CookModePage.tsx` MUST import and call `useGenerateReheatSequence` AT LEAST ONCE.** Grep assertion: `grep -c "useGenerateReheatSequence" src/pages/CookModePage.tsx` returns ≥ 2.
3. **`CookSequenceLoadingOverlay` (or whatever component name the planner chooses for the D-04 overlay) MUST be rendered conditionally on the cook-sequence `isPending` flag in `CookModePage.tsx`.** Grep assertion: `grep -E "(LoadingOverlay|isPending)" src/pages/CookModePage.tsx | grep -c "isPending"` returns ≥ 1.
4. **The `flowMode === 'reheat'` branch (currently line 360) MUST NOT render the hardcoded `reheat-1/2/3` array as the FIRST render.** The hardcoded array is the **failure fallback** (D-02) and must only execute inside an `isError`-conditional or equivalent branch. Grep assertion: `grep -B3 "reheat-1" src/pages/CookModePage.tsx | grep -cE "(isError|catch|fallback)"` returns ≥ 1 (i.e., the hardcoded array is gated by an error/fallback marker on a nearby line).
5. **The cook-sequence success branch MUST switch `flowMode` to `'cook'` (existing behaviour preserved).** Grep assertion: `grep -c "setFlowMode('cook')" src/pages/CookModePage.tsx` returns ≥ 2 (one for cook-sequence success, one for the existing `handleResumeExisting` / `handleStartCook` path; both must remain).
