## New Bugs Found (1)

### Cook Mode bootstrap fails when meal wraps a recipe (NEW)

**Symptom:** Navigating to `/cook/<meal-id>?slotId=...` (Cook Mode from plan slot) fires `GET /rest/v1/recipes?id=eq.<meal-id>` which returns 406 because `<meal-id>` is not a valid recipe_id. Result: "No steps loaded yet." Clicking "Start cook session" creates an invalid cook_session (409 on insert).

**Root cause:** `src/pages/CookModePage.tsx:85`:
```ts
const recipeIdForSteps = activeSession?.recipe_ids[0] ?? mealId
```
The fallback `?? mealId` is WRONG when `source !== 'recipe'`. When a meal wraps a recipe-type meal_item, the page should resolve the recipe ID from the meal's `meal_items` collection (look up `item_type='recipe'` rows) — NOT fall back to the mealId itself.

**Workaround:** Launch Cook Mode directly from a recipe via `/cook/<recipe-id>?source=recipe` — this path sets `isFromRecipe=true` and uses the URL param as the recipe_id correctly.

**Impact:** Blocks the "Cook this meal" button on plan slots when the slot's meal wraps a recipe. This is likely why the prior UAT (commit `dcce8f0`) used direct recipe entry and deferred the plan-slot path. The flow existed before Phase 30; not a regression from current work.

**Suggested fix:** In `CookModePage.tsx`, when `source !== 'recipe'` and no active session, resolve `recipeIdForSteps` from a `useMealItems(mealId)` query returning the first `item_type='recipe'` row's `item_id`. Current code has no hook for this.

# Pass 2 — FEED-02 bug fix + UAT

**Ran:** 2026-04-24T16:47:00Z
**Scope:** Fix FEED-02 save bug (open since Phase 20, 2026-04-22) + verify via Playwright

## Root cause

`src/hooks/useDietaryRestrictions.ts:55` used `upsert(row, { onConflict: 'household_id,member_user_id' })` but the unique index in `supabase/migrations/024_feedback_dietary.sql:64-66` is functional:

```sql
create unique index dietary_restrictions_member_unique
  on public.dietary_restrictions(
    coalesce(member_user_id::text, member_profile_id::text));
```

Postgres requires `ON CONFLICT` target columns to match an existing unique constraint/index **exactly**. A two-column composite `(household_id, member_user_id)` does not match a one-column functional expression, so every upsert returned **400**.

## Fix

Replaced upsert with a manual query-then-update-or-insert pattern. Avoids a DB migration; preserves existing RLS; no changes to the unique index (which correctly enforces "one restrictions row per user globally").

```ts
// Query first
const { data: existing } = await supabase.from('dietary_restrictions')
  .select('id').eq('household_id', ...).eq(memberColumn, ...).maybeSingle()

if (existing) {
  // UPDATE by id
  .update(row).eq('id', existing.id)
} else {
  // INSERT
  .insert(row)
}
```
